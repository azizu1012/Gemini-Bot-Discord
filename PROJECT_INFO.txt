1) PRODUCT GOALS
================================================================================

Chad Gibiti được thiết kế để:
- Trả lời hội thoại Discord theo ngữ cảnh.
- Dùng tool-calling để lấy thông tin thời gian thực.
- Duy trì memory dài hạn hữu dụng theo hướng DB-first.
- Bảo vệ ranh giới riêng tư giữa users trong môi trường multi-user.
- Cho phép admin điều tra dữ liệu chéo user theo policy đã xác thực role.


================================================================================
2) RUNTIME OVERVIEW
================================================================================

Entry points:
- main.py (thin entrypoint)
- run_bot.py (launcher chính, bot-only hoặc bot + server)
- run_bot.sh (launcher Linux/Ubuntu + PM2 modes: --pm2, --pm2-fresh)

Luồng xử lý chính nằm ở:
- src/handlers/message_handler.py

Flow tóm tắt hiện tại:
1. Nhận message (DM hoặc mention).
2. Build context/history từ DB.
3. Tính signal đa user từ DB (distinct user count + has other users).
4. Nếu admin + query cross-user detail: build evidence block từ DB.
5. Chạy reasoning loop (có thể gọi tools).
6. Chạy final synthesis (và fallback) với cùng privacy context.
7. Persist user/assistant message vào DB.

================================================================================
4) MEMORY ARCHITECTURE
================================================================================

Thành phần chính:
- src/database/repository.py
- src/managers/note_manager.py
- src/tools/tools.py (save_note/retrieve_notes bridge)

Mô hình scope:
- user: mặc định, an toàn riêng tư.
- candidate_global: fact dùng chung ở trạng thái chờ xác nhận.
- global: fact đã đủ điều kiện chia sẻ.

Policy:
- Ưu tiên retrieval theo user trước.
- Chỉ include global khi intent phù hợp.
- Chặn note mang dấu hiệu lạm dụng (impersonation/harassment/dox-like content).

Ghi chú vận hành hiện tại:
- Luồng chính đang DB-only cho history/persistence.
- JSON memory service vẫn tồn tại dạng utility legacy, không còn là hot path chính.

5) DATABASE SNAPSHOT + RESILIENCE SETTINGS
================================================================================

Bảng chính:
- messages(user_id, role, content, timestamp)
- user_notes(user_id, note_id, content, metadata, created_at, scope, importance,
             updated_at, is_active, note_type, fact_hash)

Điểm kỹ thuật:
- DB mới được bootstrap theo fresh schema khi khởi tạo.
- Schema cũ/không tương thích được detect và auto-rebuild.
- Có index phục vụ history retrieval và scoped note retrieval.

SQLite connection PRAGMAs hiện tại:
- PRAGMA journal_mode=WAL
- PRAGMA synchronous=FULL
- PRAGMA busy_timeout=2000
- PRAGMA wal_autocheckpoint=1000
- PRAGMA foreign_keys=ON

Write robustness:
- log_message_db có retry cho lỗi transient "database is locked".

Cross-user APIs đã có:
- count_distinct_message_users_db()
- has_other_users_history_db(user_id)
- search_user_messages_db(...)
  (search nội dung theo COLLATE NOCASE)

================================================================================
9) WHEN TO STAY SQLITE VS MOVE POSTGRES
================================================================================

SQLite là đủ khi:
- single-node deployment,
- traffic vừa,
- ưu tiên đơn giản vận hành.

Nên cân nhắc PostgreSQL khi:
- concurrency tăng rõ,
- cần scale nhiều process/instance,
- cần observability/transaction control cao hơn.

Khuyến nghị migration:
- Giữ interface repository ổn định.
- Chuẩn bị adapter để thay backend DB dần.
